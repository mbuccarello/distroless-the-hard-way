# SLSA Level 3: Provenance & Compliance

One of the defining tenets of "Distroless The Hard Way" is achieving and mathematically proving **Supply chain Levels for Software Artifacts (SLSA) Level 3**. 

It is not enough to say our containers are secure; we must generate a cryptographically valid artifact that allows anyone pulling our images to independently verify that the image was built natively from our source code, on our infrastructure, without tampering.

## Core Requirements of SLSA Level 3

To achieve SLSA Level 3, a repository must guarantee:
1. **Source Code in Version Control:** The application logic and build instructions are publicly verifiable.
2. **Ephemeral Build Environments:** Builds occur in isolated, temporary sandboxes.
3. **Build as Code:** The build commands are predefined configurations, not manual commands.
4. **Non-Falsifiable Provenance:** The resulting artifact contains a signature that cryptographically proves exactly what build system executed the build and from what commit.

## How We Achieve Full Compliance

We utilize a combination of GitHub Actions, Sigstore, and the `attest-build-provenance` action. 

### 1. The Build Infrastructure
All components in our stack (from `build-glibc` to `assemble-base`) run on GitHub Actions `ubuntu-latest` runners. This natively satisfies both the **Ephemeral Build Environments** and **Build as Code** requirements.

### 2. The OIDC Provider & Sigstore
Historically, signing an image required generating a Private GPG/RSA Key and storing it in GitHub Secrets. If that secret was stolen, an attacker could sign malicious binaries posing as us.

Instead, we use **Keyless Signing (Sigstore/Fulcio)**.
Our GitHub Action relies on the ephemeral `${{ secrets.GITHUB_TOKEN }}`. When our pipeline reaches the end of its build, it contacts the public Sigstore Certificate Authority (Fulcio) and requests a micro-certificate that proves: *"I am GitHub. I solemnly swear that at this exact second, branch `main` at commit `SHA` executed the file `build-glibc.yml` on my infrastructure, and it produced this exact container hash."*

### 3. The `attest-build-provenance` Action
Across all of our GitHub Action workflows, the absolute final step looks like this:
```yaml
- name: Generate SLSA Build Provenance (Level 3)
  uses: actions/attest-build-provenance@v1
  with:
    subject-name: ghcr.io/<repository>/artifacts-glibc
    push-to-registry: true
```
This action takes the ephemeral certificate generated by Sigstore and outputs a `.sig` artifact (a discrete OCI artifact directly bound to your image) containing the full Software Bill of Materials and the cryptographic signature.

## Mathematical Verification (Testing it Yourself)

You do not have to trust our documentation. You can independently verify the SLSA Level 3 integrity of any artifact generated by this repository using the official GitHub CLI.

```bash
# Verify the base image mathematically provenance back to this repository's orchestrator
gh attestation verify ghcr.io/<your-username>/base:latest -o <your-username>
```

When you run this command, it downloads the certificate from the GHCR registry, queries the public transparency log (Rekor), and confirms that the binary you are examining is untampered.
